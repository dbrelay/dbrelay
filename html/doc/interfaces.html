Interfaces

HTTP request

The SQL Bridge module should be configurable to be invoked on one or many URIs. By default it must respond to get and post requests at the /sql path.

HTTP request carries information in the HTTP header and parameter key=value pairs. The application must not expect request information in HTTP headers. Only request parameters must be used. All defaults are configured in the build header files. The parameters below are grouped for the clarity:

Database Server:

Name Description
sql_dbtype Database driver name in viaduct. Currently, «mssql» or «mysql» . The «mssql» value is default, if the field is missing.
sql_server Hostname on which the SQL server is running.
sql_port Optional port number, on which the SQL server is listening. 1433 is the default.
sql_database Optional name of the primary database for the connection. User's default database is used, if not specified.
sql_user Username string recognised by the SQL server.
sql_password Optional password for the sql_user.

 

Connection Lifecycle:

Name Description
connection_name Optional name to persist this connection under.
connection_timeout Optional number of seconds from the response to the last query before the connection will be considered idle and marked for destroying. The default is 60. The silently enforced maximum connection lifespan is 28800 (8 hours).
http_keepalive Optional flag to support broken HTTP clients. When set to 0 Viaduct will close an HTTP 1.1 connection after returning the response. Has no effect on HTTP 1.0 connections.
query_tag Optional tag which will be returned verbatim with the response. Used to help client to identify responses to asynchronous requests.
log_level Optional log level setting for the whole application, connection or the specific query.
log_level_scope Optional value of either «server», «connection», or «query» value which designated the scope of applying the specified scope_level. By default it is assumed to be «query». When the scope is set to «connection», then all connections, matching the «sql_server», «sql_port», «sql_database», and «sql_user» request parameters, current or new, log according to the specified level. An absent optional parameter acts as a wildcard. Setting the «server» scope resets all the «connection»-specific log levels.

 

SQL query:

Name Description
sql Optional TSQL statement batch conformant with MS SQL Server 2005 TSQL. "Sql" and "param[]" parameters may be omitted when, for example, setting the new application log level.
paramX Optional postfix-numbered set of colon-separated pairs «type»:«value» of positional SQL parameters to be bound into the SQL query. Output parameters are not supported. Parameters should come as an integer-indexed zero-based sequence. Gaps in parameter indexes should cause notice-level errors. SQL parameter values are expected to be URL-encoded. The accepted «type» values are int, smallint, tinyint, numeric, float, real, char, varchar, datetime, smalldatetime.
error_style Optionally either "http" or "json", specifying either to return HTTP 5xx reply, or JSON response with error in the "log" JSON object. The default is "json".

 

Administrative Requests:

Name Description
admin A keyword to get to the set of administrative commands. When specified, only administrative request's parameters are taken in consideration. A specific administrative parameter is expected in the same http request.
status Request the server status as a part of the administrative http request.
cmd Execute a command on the sever. Specific commands and their format to be decided later in the project.

Log and PID Files

If no directories specified, logs and PID files are created in system-vendor recommended directory as defined in the application source header files. File names, if configurable in Nginx, should include the executable name, listening HTTP port, date and time stamp and, appropriately, have .log or .pid extension. The PID file should only contain a numeric PID of the application process. Content of the log files at appropriate levels will be decided later in the project. If using the system-vendor recommended directories fails, then log and PID files should go into a temporary directory together with run time files.

Log files are written according to the «log_level» setting in the syslog format as per §4.1 of RFC 3164. The messages are assigned to «user-level messages» facility (value 1). The following are the meanings for the «log_level» values, mapped to the syslog «severity» levels:

Name Description
debug The complete http connection, request and database server lifecycle information, including full set of timings on both client and database sides.
informational Live configuration changes. For now, this is only «log_level» for «server» and «connection» scopes.
notice All query errors together with related requests' data, parameters, and client's data: ip address, http headers, excluding passwords.
warning

Events, which may indicate misconfiguration or remote error condition, but do not threaten the Viaduct server integrity. Most events, which are likely to impact multiple clients. Connection and request records should include relevant data mentioned above. Passwords should not be logged. Example events:

  • Timeouts on database server queries.
  • Malformed requests.
  • Errors establishing connections.
  • Connections re-established more often than once every three seconds.
error

Unsolicited database server side disconnects. Provides:

  • The affected connection data with all its parameters, excluding password. Include connection remote IP address, time established, requests per hour, data in and data out.
  • Active and pending requests per each connection, with all the requests' parameters and client's data: ip address, http headers, excluding passwords.
critical

Log only when Viaduct server can not continue and will shutdown. the following data is reported:

  • List of open connections with all their parameters, excluding passwords. Include connection remote IP addresses, time established, requests per hour, data in and data out.
  • Active and pending requests per each connection, with all the requests' parameters and client data: ip address, http headers, excluding passwords.
  • Server process stats - PID, UID, GID, memory sizes, configs, whatever else developer deems reasonable.

Static Files

Examples of static files are HTML, JavaScript, CSS, SWF, Image and other files, which are sent to HTTP clients straight from the server's filesystem without modifications.

If no --root directory specified, static files are searched for in system-vendor recommended directory as defined in the application source header files.

Run time files

The web server needs a place for temporary files, currently residing in the client_body_temp, fastcgi_temp, and proxy_temp subdirectories of the build's PREFIX directory. If those do not exist per the build-time configuration, the server will create them in a single temporary directory. The directory should be removed during the graceful process shutdown. No cleanup attempted in the case of abnormal shutdown. Name of the temporary directory should be available via a management web page.

Command line Parameters

The application must have defaults for everything, so all command line parameters are optional.

Param Value Default Description
--port integer 80 A TCP port number for the Viaduct's HTTP server to listen on.
--root directory hardcoded per build target in header files A directory for the Viaduct's HTTP server to serve the static files from.
--config file name hardcoded per build target in header files A file name for run-time configuration to load.
--log-dir directory name hardcoded per build target in header files A directory name where the log files should go.
--pid-file file name hardcoded per build target in header files A file name for the PID file.
